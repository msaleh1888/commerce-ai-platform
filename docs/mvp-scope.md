# MVP Scope

## Objective

Build a polished portfolio MVP that demonstrates a realistic commerce AI operations workflow end to end. The MVP should be smaller than the full blueprint, but it must feel complete, professional, and credible.

## MVP Product Story

A catalog manager at a retailer imports supplier catalog files. The platform validates and normalizes the records, indexes them for search, identifies likely duplicates, and shows reviewable evidence. The manager can search the catalog, inspect matching candidates, approve or reject a merge proposal, and see evaluation and audit evidence.

## In Scope

### Identity and Tenancy

- Email/password authentication suitable for local/demo use.
- Tenant creation or seeded demo tenants.
- Memberships and roles.
- Tenant-scoped API access.
- Role-based permissions for read, import, review, approve, and admin actions.

### Catalog Ingestion

- CSV upload.
- JSON upload.
- Import records stored with original source payloads.
- Import state machine.
- Validation errors stored per row.
- Idempotency by tenant, source, file hash, and import request.
- Retry-safe worker tasks.
- Progress status visible through API and UI.

### Product Model

- Supplier products.
- Canonical products.
- Brands, categories, normalized attributes, price, currency, identifiers.
- Linkage from supplier record to canonical product.
- Product lifecycle status.

### Search and Retrieval

- Lexical baseline search.
- Dense vector search.
- Hybrid retrieval with simple result fusion.
- Metadata filters for tenant, category, brand, price, and status.
- Search result explanations that cite product fields or retrieval units.
- Search evaluation report with frozen query set.

### Duplicate Detection and Review

- Candidate duplicate generation using identifiers, normalized title similarity, brand, category, and vector similarity.
- Review cases with evidence.
- Confidence score and reason codes.
- Human decision: merge, keep separate, mark as variant, or defer.
- Approved operation updates authoritative PostgreSQL state.
- Rejected/deferred operations remain visible.

### Audit and Safety

- Audit events for imports, search evaluations, review decisions, and catalog mutations.
- Tenant isolation tests for API and retrieval.
- Prompt-injection test fixtures for product content, even if LLM generation is minimal in MVP.
- AI/model outputs treated as untrusted structured inputs.

### Evaluation

- Small deterministic test fixtures for CI.
- Reproducible benchmark manifests.
- Retrieval metrics: nDCG@10, MRR, Recall@10, Recall@50, latency.
- Matching metrics: precision, recall, F1, false-merge rate, review workload.
- Baseline comparison tables.

### Frontend

- App shell with tenant context.
- Import screen.
- Import status screen.
- Product catalog browser.
- Product search page.
- Duplicate review queue.
- Review case detail page.
- Evaluation summary page.
- Audit history page.

## Out of Scope for MVP

- Real payment, checkout, or order workflows.
- Real Shopify integration.
- MCP server.
- LangGraph ReAct agent.
- Long-term memory.
- OpenAI Agents SDK comparison.
- Kubernetes deployment.
- Full Prometheus/Grafana stack.
- Large-scale load testing beyond local repeatable checks.
- Real-time collaborative review.
- Advanced personalization.

## Technical Scope

### Required

- FastAPI backend.
- PostgreSQL source of truth.
- Qdrant vector index.
- Redis and Celery for async processing.
- Next.js frontend.
- Docker Compose local environment.
- GitHub Actions CI skeleton.
- pytest unit and integration tests.
- Playwright smoke test for the main demo path.

### Deferred

- Kubernetes manifests.
- Full OpenTelemetry collector stack.
- Langfuse integration.
- Multi-provider model abstraction.
- Advanced reranking provider.

## MVP Exit Criteria

- A fresh local environment starts from documented commands.
- A demo tenant can import a catalog and complete processing.
- Search returns only tenant-authorized products.
- At least two retrieval configurations can be compared.
- Duplicate review cases can be approved or rejected.
- Approved catalog decisions are idempotent and audited.
- CI runs formatting, unit tests, selected integration tests, and a small evaluation smoke test.
- README can truthfully link to measured results generated by the code.

## Quality Bar

The MVP does not need every advanced feature from the blueprint. It does need to be coherent, tested, and presentable. A small complete workflow with evidence is better than a large unfinished platform.
