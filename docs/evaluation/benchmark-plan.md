# Benchmark Plan

## Purpose

This plan defines how the MVP will produce credible evidence for retrieval and product-matching quality. It complements the broader real-data testing plan and starts with small, reproducible subsets before scaling to larger public datasets.

## Principles

- Every advanced configuration must be compared with a simple baseline.
- Dataset versions and splits must be frozen before comparing results.
- Retrieval, generation, matching, and agent behavior must be evaluated separately.
- Public claims must distinguish planned targets from achieved measurements.
- False product merges are more dangerous than missed matches.

## MVP Evaluation Scope

### Retrieval

Compare:

- Lexical baseline.
- Dense vector retrieval.
- Hybrid lexical + dense retrieval.

Measure:

- nDCG@10.
- MRR.
- Recall@10.
- Recall@50.
- Precision@10 where labels are available.
- Empty-result rate.
- Duplicate-result rate.
- p50 and p95 latency.

### Product Matching

Compare:

- Exact identifier baseline.
- Deterministic normalized-title/brand/category matching.
- Hybrid candidate scoring with vector similarity.

Measure:

- Precision.
- Recall.
- F1.
- False-merge rate.
- Missed-match rate.
- Review workload by confidence threshold.
- Candidate-generation recall.

### Tenant Isolation

Measure:

- Cross-tenant API records returned: must be 0.
- Cross-tenant retrieval records returned: must be 0.
- Cross-tenant mutations allowed: must be 0.

## Starting Datasets

### Fixture Dataset

Use a small committed fixture dataset for CI and deterministic development.

Contents:

- 50 to 200 products.
- 2 tenants.
- 3 to 5 categories.
- Intentional duplicate supplier records.
- Intentional variants.
- Malformed rows.
- Search queries with expected relevant product IDs.
- Matching pairs labeled match, non-match, and variant.

Purpose:

- Fast CI.
- Deterministic behavior.
- Tenant isolation tests.
- Import and review workflow tests.

### Amazon ESCI Subset

Use an English subset for retrieval evaluation once the fixture harness works.

Purpose:

- Product search relevance.
- Lexical vs dense vs hybrid comparison.
- Query ambiguity and substitute/complement behavior.

### WDC Products Subset

Use a manageable subset for matching and duplicate detection once review workflow exists.

Purpose:

- Entity resolution.
- False-merge measurement.
- Review threshold calibration.

### Coveo Dataset Later

Use the Coveo SIGIR E-Commerce dataset after MVP foundations are stable.

Purpose:

- End-to-end behavioral replay.
- Larger catalog ingestion.
- Session/query/event evaluation.

## Required Reports

### Retrieval Report

Include:

- Dataset and manifest.
- Configurations compared.
- Metric table.
- Latency table.
- Query examples where hybrid improved results.
- Query examples where hybrid failed.
- Decision on whether the added complexity is justified.

### Matching Report

Include:

- Dataset and manifest.
- Candidate-generation approach.
- Precision/recall/F1.
- False-merge rate.
- Review workload by threshold.
- Examples of safe matches, unsafe matches, and missed matches.
- Recommended threshold for review routing.

### Security Report

Include:

- Tenant isolation test count.
- Cross-tenant API failures.
- Cross-tenant retrieval failures.
- Unauthorized mutation attempts.
- Prompt-injection fixture outcomes.

## CI Strategy

Pull request CI should run:

- Fixture import test.
- Tenant isolation test.
- Small retrieval evaluation smoke test.
- Small matching evaluation smoke test.
- Idempotent approval test.

Longer local or scheduled runs can use:

- Larger ESCI subset.
- Larger WDC subset.
- Provider-backed embeddings.
- Latency and throughput checks.

## Publication Standard

The README may publish only results generated by committed code and versioned manifests. Any future desired metrics must be labeled as targets or planned work.
